## 1. Текущее поведение `hud.py` (зафиксировано статическим анализом)

### Что делает

- Открывает камеру 0 (`cv2.VideoCapture(0)`).
- В цикле читает кадр → вычисляет `dt = curr_time - prev_time` → рисует HUD-панель → показывает в окне `'Smart Glasses Simulation'`.
- Выход: `q`.

### Вшитые "фичи" (текущие модули в плоском коде)

| Фича | Реализовано как |
|---|---|
| Статусная подложка | `cv2.rectangle` чёрный прямоугольник (20,20)→(300,100) |
| Текст `SYSTEM: AR_MODE_ON` | `cv2.putText` (зелёный) |
| Счётчик FPS | `cv2.putText` (жёлтый), использует `dt` |
| Захват видео | `cv2.VideoCapture(0)` напрямую |
| Вывод | `cv2.imshow` напрямую |

### Известные баги (найдены статически)

| # | Баг | Где | Последствие |
|---|---|---|---|
| 1 | `NameError` на 1-м кадре | строка 13 (`f"FPS: {int(1/dt)}"`) вызывается до присвоения `dt` в строках 24-26 | Падение на первом кадре при запуске |
| 2 | `prev_time = 0` на старте | строка 17 | Первый `dt` = `time.time()` ≈ 1.7e9 → гигантский FPS в логе, деление на ~0 (см. баг 1) |
| 3 | Жёсткая привязка к `cv2.VideoCapture(0)` | строка 16 | Невозможно запустить на видеофайле / RTSP без правки кода |
| 4 | Жёсткая привязка к `cv2.imshow` | строка 31 | Невозможно вывести картинку куда-либо ещё (фреймбуфер AR, сетевой стрим) |
| 5 | `time.time()` напрямую | строка 24 | wall-clock чувствителен к смене системного времени; подмешивается в dt |
| 6 | Окно создаётся внутри цикла | строка 31 каждый кадр пересоздаёт handle окна (на самом деле нет, но имя окна жёсткое) | Минорная проблема дизайна |
| 7 | Никакого graceful shutdown | `cap.release()` + `destroyAllWindows()` без `try/finally` | При исключении камера не освобождается |

### Карта вызовов `time.*`

| Файл | Строка | Вызов | Назначение |
|---|---|---|---|
| `hud.py` | 17 | `prev_time = 0` (инициализация, не вызов, но источник бага 2) | Заготовка под первый dt |
| `hud.py` | 24 | `time.time()` | Единственный замер времени в проекте — закрыть в Phase 1 через `Clock` |

**Итог:** один источник `time.time()` в проекте. Карта рефакторинга тривиальна — `Clock.tick()` будет единственной точкой замера.

## 2. Smoke-test 50 кадров

Запуск в живой среде невозможен: физической веб-камеры в среде нет, ffmpeg в PATH не обнаружен, сгенерировать видеофайл без зависимостей и камеры тоже нечем.

**Решение для Phase 0:** зафиксировать ожидаемое поведение на уровне кода — первый кадр даст `NameError` (см. баг 1). Smoke-test в его настоящем виде будет возможен только после Phase 2, когда появится `VideoFileSource`. Тогда же проверим 50 кадров против синтетического видео (numpy + `cv2.VideoWriter`).

**Что фиксируем как baseline:** `hud.py` как есть не запустится ни на одном живом кадре. Это ожидаемо — это и есть повод для рефакторинга.

## 3. Список "модулей"/фич, вшитых в текущий скрипт

Для переноса в `src/modules/` в Phase 3:

1. **Status panel** — прямоугольник + текст `SYSTEM: AR_MODE_ON` → кандидат на `SystemStatusModule`.
2. **FPS counter** — счётчик на базе `dt` → кандидат на отдельный `FpsCounterModule` или часть `SystemMonitorModule`.
3. **Future: clock overlay** — часы реального времени (в hud.py нет, но в плане есть как `clock_module.py`).
4. **Future: system monitor** — CPU/RAM (в hud.py нет).
5. **Future: weather** — внешний API (в hud.py нет).
6. **Future: face detection** — CV-модуль (в hud.py нет).

## 4. Что блокирует Phase 1

Ничего. Phase 1 (core: Frame, ICameraSource, IHUDModule, Clock) — чистый код, не зависит ни от камеры, ни от видеофайлов, ни от cv2 в `update`/`render` интерфейсах (только в `Frame.image: np.ndarray` — допустимая зависимость от типа).
